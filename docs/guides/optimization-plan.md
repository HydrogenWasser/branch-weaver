# Branch Weaver 优化计划

## 1. 文档目标

本文档用于指导 Branch Weaver 的后续优化工作，目标不是一次性大改，而是将当前已经确认的问题拆分为可以逐步实施、逐步验证的工程任务。

本文档重点覆盖以下七个维度：

- 类型安全
- 代码风格
- 组件设计
- 代码质量
- 可维护性
- 可扩展性
- 性能

本文档默认遵循以下原则：

- 先修复数据正确性问题，再做结构性重构。
- 先统一领域规则，再优化 UI 组织和性能路径。
- 每个阶段都必须能独立交付，避免“大重构期间长期不可用”。
- 所有优化都以当前项目的产品边界为前提：浏览器内单页应用、无后端、文件通过浏览器导入导出。

## 2. 当前状态总结

### 2.1 当前优势

- 项目的领域模型已经比较完整，支持全局变量、条件可见性、条件路由、效果应用、导入导出与迁移。
- `lib/`、`store/`、`components/`、`types/` 的基础分层方向正确。
- `validateStoryProject()`、`parseProjectJson()`、`exportValidationErrors()` 形成了相对完整的运行时校验链。
- Zustand store 使用 `withProjectMutation()` 统一管理历史记录、脏状态和错误清理，基本状态纪律较好。
- 已经具备基础性能意识，例如 `useDeferredValue` 和 React Flow 的 `onlyRenderVisibleElements`。

### 2.2 当前主要问题

#### P0 正确性问题

- `migrateProject()` 在导入项目时会无条件覆盖 `fileTriggers`，会造成已有字段丢失。
- `PreviewPlayer` 的回退逻辑只回退节点历史，不回退运行时全局变量状态，导致试玩结果不可信。
- “至少保留一个节点”的约束只存在于部分 UI 入口，store 自身没有完整守住这个不变量。

#### P1 类型与规则问题

- `StoryCondition` 和 `StoryEffect` 的值类型是宽联合，许多业务约束只能依赖运行时校正，而不是编译期约束。
- `StoryGlobal` 的 Zod schema 不是按 `valueType` 判别的联合，`defaultValue` 与 `valueType` 的关系不够严格。
- `validateStoryProject()` 和 `exportValidationErrors()` 存在规则重复实现，后续修改容易发生漂移。

#### P2 结构与维护问题

- `App.tsx` 承担了过多职责：文件操作、快捷键、搜索、布局、面板排序、弹窗开关、错误展示。
- `ChoicesEditor.tsx` 同时承载可见性、效果、路由三套复杂编辑逻辑，体积偏大。
- `editorStore.ts` 已经成为超大单文件，继续扩展会导致改动范围越来越宽。
- 面板 ID、localStorage key、窗口键盘监听等模式在多个组件内重复出现。

#### P3 性能与扩展问题

- 所有 mutation 都走整项目深拷贝和完整历史快照，文本编辑和频繁操作的成本偏高。
- `CanvasGraph` 每次 project 变化都会重建整张图的节点和边，局部编辑会触发全量派生。
- `StoryNodeCard` 通过 `JSON.stringify(data.storyNode.choices)` 驱动内部更新，开销不稳定。
- 搜索索引、导出检查和多个派生 Map 挂在顶层渲染链上，后续规模增长时会逐步放大成本。

## 3. 总体优化目标

本轮优化不追求引入新的产品能力，而是为后续功能开发建立更稳定的基础。

优化完成后的目标状态如下：

- 数据正确性问题全部消除，导入、导出、试玩行为一致可信。
- 领域规则在类型层和运行时层都更一致，减少“类型能通过、运行时再兜底”的情况。
- store 和大组件职责收敛，新增字段或新编辑能力时改动面更可控。
- 性能瓶颈从“当前就容易累积的问题”变为“仅在大规模故事图下才值得进一步优化的问题”。
- 每一阶段都有明确验收标准，避免优化过程失控。

## 4. 优先级原则

本计划采用如下优先级顺序：

1. 数据正确性与用户可感知错误
2. 类型安全与规则一致性
3. 结构重构与可维护性
4. 性能止损与渲染优化
5. 文档补全与长期工程化整理

出现冲突时，按以下规则处理：

- 如果一个优化会改变用户数据语义，则必须先做验证链和迁移策略。
- 如果一个性能优化会引入更高的维护复杂度，则优先做低风险止损，不急于做复杂抽象。
- 如果一个结构重构无法在单阶段内保持行为等价，则需要拆小。

## 5. 分阶段优化路线图

## Phase 0：正确性与稳定性修复

### 5.1 目标

先修复已经确认的行为错误和数据丢失问题，确保现有功能可信可用。

### 5.2 范围

- `src/lib/story.ts`
- `src/components/PreviewPlayer.tsx`
- `src/store/editorStore.ts`
- `src/components/DraggablePanel.tsx`

### 5.3 具体任务

#### 任务 0.1：修复导入时覆盖 `fileTriggers` 的问题

问题位置：

- `src/lib/story.ts` 中 `migrateProject()`

执行内容：

- 在迁移 v2 项目时保留已有的 `fileTriggers`。
- 在迁移 v1 项目时为缺失字段补默认值 `[]`。
- 复核 `normalizeNode()` 是否仍然能正确处理空值、重复值和空字符串。

预期收益：

- 避免用户导入后再保存时丢失已有数据。

验收标准：

- 打开包含 `fileTriggers` 的 v2 JSON 后，保存导出的结果仍保留这些值。
- 打开 v1 JSON 时仍能正确迁移。

#### 任务 0.2：修复试玩预览回退不回滚全局变量的问题

问题位置：

- `src/components/PreviewPlayer.tsx`

执行内容：

- 将 `history` 从 `string[]` 改为帧栈结构，例如：
  - `type PreviewFrame = { nodeId: string; globals: Map<string, boolean | number> }`
- 前进时同时记录目标节点和应用 effects 后的 globals 快照。
- 回退时恢复上一帧，而不是仅修改节点路径。
- 重新检查重开、关闭、重新打开预览时的初始化逻辑。

预期收益：

- 试玩预览的行为与实际分支逻辑一致，用户能依赖预览验证剧情。

验收标准：

- 含 effects 的 choice 被点击后，前进和回退都会得到一致的 globals 状态。
- Restart 可以恢复到初始 globals。

#### 任务 0.3：在 store 层补齐关键不变量

问题位置：

- `src/store/editorStore.ts`

执行内容：

- 在 `removeNode()` 内部增加“至少保留一个节点”的保护，不只依赖 `deleteSelection()`。
- 复核与开始节点相关的逻辑，确保删除开始节点时 fallback 行为始终合法。
- 检查其他可能依赖 UI 前置限制的入口，补齐 store 层硬约束。

预期收益：

- 后续即使更换 UI 入口或新增命令，底层状态仍保持合法。

验收标准：

- 任意路径都不能生成“零节点项目”。
- 删除开始节点后，新的开始节点和 `Start` tag 始终一致。

#### 任务 0.4：修复低成本但明显的质量问题

问题位置：

- `src/components/DraggablePanel.tsx`

执行内容：

- 修复拖拽把手字符乱码，改成稳定的 ASCII 或明确可用的字符。
- 顺手检查其他明显编码问题或展示异常。

预期收益：

- 提升界面完整性，避免低级质量瑕疵影响整体观感。

验收标准：

- 拖拽把手在本地开发与生产构建中显示正常。

### 5.4 风险

- `PreviewPlayer` 的历史结构修改会影响若干事件处理，容易引入状态同步错误。
- store 不变量增强后，可能暴露出某些 UI 逻辑原本依赖非法状态的隐性假设。

### 5.5 阶段验收

- `npm run build` 通过。
- 手工验证导入/保存、预览前进/后退、删除节点、拖拽面板。
- 不出现字段丢失、预览状态错误或最后节点被删除的情况。

## Phase 1：类型安全与规则统一

### 6.1 目标

把当前主要依赖运行时校正的领域约束，尽可能前移到类型和 schema 层；同时统一规则来源，降低未来维护成本。

### 6.2 范围

- `src/types/story.ts`
- `src/lib/story.ts`
- `src/lib/conditions.ts`
- `src/components/ConditionEditor.tsx`
- `src/components/ChoicesEditor.tsx`

### 6.3 具体任务

#### 任务 1.1：将 `StoryGlobal` schema 改为判别联合

执行内容：

- 把当前的 `globalSchema` 改成按 `valueType` 判别的 Zod union。
- `boolean` 分支要求 `defaultValue: boolean`。
- `number` 分支要求 `defaultValue: number` 且为有限值。

预期收益：

- schema 与 TypeScript 类型表达更一致。
- 导入时能更早发现不合法数据。

#### 任务 1.2：收紧 `StoryCondition` 与 `StoryEffect` 的领域表达

执行内容：

- 评估是否将 `StoryCondition` 和 `StoryEffect` 建模为更明确的判别联合。
- 至少要把“布尔全局只允许 `eq`/`neq`”“布尔 effect 只允许 `set`”体现到更靠近类型定义的位置。
- 减少编辑器层到处使用 `boolean | number` 和 `as` 断言的情况。

预期收益：

- 让更多错误在编译时暴露，而不是依赖 normalize 和 export validation 兜底。

#### 任务 1.3：统一验证规则来源

执行内容：

- 识别 `validateStoryProject()` 与 `exportValidationErrors()` 中重复的规则。
- 抽取共享规则函数，例如：
  - 条件合法性
  - effect 合法性
  - route target 合法性
  - start node / start tag 一致性
- 区分“抛错式导入校验”和“列表式导出校验”的输出方式，但尽量共用底层规则判断。

预期收益：

- 避免后续加新规则时一处修改、一处遗漏。

#### 任务 1.4：收紧 UI 层的字面量类型

执行内容：

- 为 sidebar / inspector 面板 ID 建立字面量联合类型。
- 为路由模式、effect 操作符等 UI 中重复出现的字符串建立统一来源。
- 减少组件内散落的 `"direct" | "conditional"`、字符串 key 和 `as` 断言。

预期收益：

- 降低拼写错误和局部漂移的风险。

### 6.4 风险

- 类型收紧可能牵动多个编辑器组件，需要逐步改，不适合一次性大爆改。
- 如果判别联合设计过重，可能增加局部开发复杂度，需要平衡易用性。

### 6.5 阶段验收

- `npm run build` 通过。
- 导入 v1 / v2 项目仍正常。
- 条件编辑、effects 编辑、导出校验结果与修复前保持语义一致。
- `as` 断言数量明显下降，宽联合传播范围收窄。

## Phase 2：结构重构与可维护性提升

### 7.1 目标

降低单文件复杂度，收窄职责边界，让后续新增能力时改动面更可控。

### 7.2 范围

- `src/App.tsx`
- `src/components/ChoicesEditor.tsx`
- `src/components/Inspector.tsx`
- `src/store/editorStore.ts`
- 新增 hooks 或 store slice 文件

### 7.3 具体任务

#### 任务 2.1：拆分 `App.tsx`

执行内容：

- 提取文件操作相关逻辑，例如 `useProjectFileActions`。
- 提取全局快捷键逻辑，例如 `useEditorShortcuts`。
- 提取侧边栏面板编排和 localStorage 持久化逻辑，例如 `usePanelOrder`。
- 保持 `App.tsx` 专注于高层布局和组合。

预期收益：

- 降低根组件心智负担。
- 后续修改文件行为或快捷键时不再同时触碰主界面结构。

#### 任务 2.2：拆分 `ChoicesEditor.tsx`

执行内容：

- 将当前编辑器分为三个明确区块：
  - 可见性编辑
  - effects 编辑
  - 路由编辑
- 为每一块定义最小 props，避免子组件继续读取整个 project。
- 梳理重复的 `nodeId/choiceId` 构造和事件处理。

预期收益：

- 选择编辑器更容易阅读、复用和测试。
- 新增 effect/operator 时更容易定位影响点。

#### 任务 2.3：收敛 `Inspector.tsx` 的职责

执行内容：

- 将节点基础信息、标签、文件触发器、颜色、选项摘要分成独立渲染片段或小组件。
- 把 inspector 内部的面板排序与 localStorage 管理提取为复用逻辑。

预期收益：

- 降低单组件体积。
- inspector 的后续扩展成本下降。

#### 任务 2.4：拆分 `editorStore.ts`

执行内容：

- 评估按功能拆分为多个 slice：
  - project 基础管理
  - globals
  - nodes
  - choices
  - history / dirty / error
- 至少把重复的 mutation 模式和辅助逻辑收敛，不再让所有行为堆在一个文件中。
- 保持“唯一数据源仍是 store”，但减少单文件规模和认知负担。

预期收益：

- 后续新增字段或行为时更容易定位改动位置。
- 更适合后续引入更细粒度 selector。

### 7.4 风险

- 结构重构虽然不改变产品行为，但容易引入小型回归。
- 如果同时拆组件和拆 store，建议分成多个小提交，不要一次完成。

### 7.5 阶段验收

- 根组件、store、choices 编辑器的文件体积和复杂度明显下降。
- 现有行为不变，构建通过。
- 手工验证以下场景：
  - 新建项目
  - 打开/保存
  - 添加/删除节点与选项
  - 编辑条件与 effects
  - inspector 排序与侧栏排序持久化

## Phase 3：性能止损与渲染优化

### 8.1 目标

优先消除当前已知的高频无谓重算，提升中大型故事图下的可编辑性，但不急于引入过于复杂的性能基础设施。

### 8.2 范围

- `src/store/editorStore.ts`
- `src/App.tsx`
- `src/components/CanvasGraph.tsx`
- `src/components/StoryNodeCard.tsx`
- `src/lib/search.ts`

### 8.3 具体任务

#### 任务 3.1：优化历史系统

执行内容：

- 先做低风险止损：
  - 文本输入是否可以改为 debounce 提交或 blur 提交再写历史。
  - 节点拖拽是否可以批量合并历史记录。
- 中期方案再评估 patch-based history 或 command-based history。
- 明确哪些操作需要即时历史，哪些操作可以合并。

预期收益：

- 降低高频输入场景下的整项目深拷贝成本。
- 降低历史栈带来的内存增长速度。

#### 任务 3.2：优化画布派生数据

执行内容：

- 收窄 `CanvasGraph` 的 store 订阅范围，不再直接依赖整个 `project` 做全量派生。
- 评估是否移除重复的 `canvasNodes` state，减少同步成本。
- 将节点和边的生成逻辑整理成更稳定的 selector 或派生层。

预期收益：

- 选择变化、文本变化时减少整图重建。
- 提升大型故事图下的交互响应。

#### 任务 3.3：收窄 `StoryNodeCard` 的内部更新触发条件

执行内容：

- 去掉 `JSON.stringify(data.storyNode.choices)` 作为 effect 依赖。
- 改用更明确的尺寸相关依赖，例如：
  - 标题
  - 正文
  - tags
  - choices 数量
  - 条件摘要计数
- 如果确实需要更稳定的签名，构造更轻量的尺寸摘要字段，而不是整数组序列化。

预期收益：

- 降低节点多、选项多时的重复序列化开销。

#### 任务 3.4：解耦搜索与导出检查的根渲染路径

执行内容：

- 搜索索引尽量只依赖搜索相关字段，而不是整个 project 对象。
- `exportValidationErrors()` 改为：
  - 保存前必跑
  - Export Checks 面板可见时再跑
  - 或者用受控缓存，而不是每次根渲染都重新计算

预期收益：

- 减少顶层渲染时的无谓计算叠加。

### 8.4 风险

- 性能优化最容易引入“行为没有变，但依赖更新时机变了”的隐性回归。
- 不建议在类型、结构尚不稳定时同步做大规模性能重写。

### 8.5 阶段验收

- 在中等规模故事图下，文本输入、节点选择、拖拽移动的体感明显更平滑。
- 关键功能保持正确：
  - 画布渲染
  - 选项连线
  - 搜索高亮
  - 自动布局
- 构建通过。

## Phase 4：文档补全与工程化整理

### 9.1 目标

在前面几个阶段完成后，把已形成的规则、约束和优化结果文档化，降低后续回归风险。

### 9.2 范围

- `README.md`
- `docs/guides/`
- 必要时补充新的开发说明文档

### 9.3 具体任务

#### 任务 4.1：补充类型与数据模型说明

- 说明 v2 数据结构、迁移边界、字段约束。
- 记录全局变量、条件、effects、路由模式的规则。

#### 任务 4.2：补充 store 设计说明

- 说明 history、dirty、selection 的管理方式。
- 如果已拆 slice，记录新的组织方式。

#### 任务 4.3：补充性能注意事项

- 记录哪些 selector、派生计算、组件 memo 化是有意设计。
- 明确后续新增功能时要避免的模式。

### 9.4 阶段验收

- 文档能帮助新开发者理解项目结构和约束。
- 优化后的关键工程决策有文字落地，不依赖口头记忆。

## 6. 推荐执行顺序

推荐按以下顺序逐步实施：

1. Phase 0 全部完成
2. Phase 1 的规则统一与 schema 收紧
3. Phase 2 的组件拆分
4. Phase 2 的 store 拆分
5. Phase 3 的历史系统止损
6. Phase 3 的画布渲染优化
7. Phase 4 的文档整理

原因如下：

- Phase 0 风险低、收益高，而且其中的问题会直接影响用户数据或预览可信度。
- Phase 1 完成后，结构重构时会更清楚哪些规则应该保留在领域层。
- Phase 2 做完后，Phase 3 的性能优化才不会在混乱结构上反复返工。

## 7. 建议的任务拆分粒度

为了控制回归风险，建议每次实施只选择一个小批次，不要一次性全做。

推荐拆分方式如下：

- 批次 A：`fileTriggers` 导入修复 + 拖拽把手乱码修复
- 批次 B：`PreviewPlayer` 历史帧修复
- 批次 C：store 不变量补齐
- 批次 D：`StoryGlobal` schema 判别联合
- 批次 E：验证规则共享化
- 批次 F：拆 `App.tsx`
- 批次 G：拆 `ChoicesEditor.tsx`
- 批次 H：拆 `editorStore.ts`
- 批次 I：历史系统止损
- 批次 J：画布派生数据优化

## 8. 手工验证清单

由于当前项目没有测试套件，优化阶段必须使用统一的手工验证清单。

每一批次至少检查以下项目：

- 能否正常启动开发服务器
- `npm run build` 是否通过
- 新建项目是否正常
- 示例项目是否能正常加载
- JSON 文件是否能正常打开
- JSON 文件是否能正常导出
- 开始节点和 `Start` tag 是否始终一致
- 节点能否新增、删除、拖拽、自动布局
- 选项能否新增、删除、连线、修改目标
- 条件可见性与条件路由是否正常编辑
- effects 是否能正确应用于试玩预览
- 试玩预览能否正确前进、后退、重开
- 搜索结果与节点高亮是否正常
- 侧栏和 inspector 面板排序是否能正确持久化

## 9. 暂不纳入本轮的事项

以下方向有价值，但不建议在本轮优先处理：

- 引入完整测试框架
- 引入 Web Worker 做布局或搜索异步化
- 引入更复杂的持久化层
- 重做视觉设计系统
- 新增大范围产品功能，例如字符串全局变量、脚本化条件表达式、批量节点模板

这些工作应当在前述基础问题收敛后再考虑。

## 10. 成功标准

如果本计划按顺序实施完成，可以认为优化达标：

- 已知正确性问题全部消除。
- 关键业务约束不再主要依赖运行时兜底。
- 根组件、复杂编辑器和 store 的职责边界明显改善。
- 中等规模项目下的交互体验稳定，没有明显卡顿。
- 新增一个字段或一类编辑能力时，影响面能够被限定在少量文件内。
- 核心设计决策已经写入文档，后续维护不再依赖隐性知识。

## 11. 后续执行建议

推荐采用以下协作方式推进：

- 每次只选一个批次执行。
- 每个批次先给出具体实施计划，再开始修改。
- 每个批次完成后立即做一次构建验证和手工回归。
- 如果某一阶段中途发现需要更大范围重构，优先回到文档更新计划，而不是直接扩散改动范围。

这样可以保持项目持续可用，并让每次优化都具备明确边界和可验证结果。
